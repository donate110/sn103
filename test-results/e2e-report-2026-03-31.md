## Recommendations

### 1. Fix /api/network/status cold start (Priority: HIGH)
The 5.7s cold start is the worst in the system. Options:
- Add a Vercel cron job (`vercel.json` cron) to hit this endpoint every 5 minutes
- Pre-compute and cache metagraph data at the edge
- Add a loading skeleton to /network page that shows immediately

### 2. Keep serverless functions warm (Priority: HIGH)
The pattern across every endpoint is "first call slow, subsequent calls fast." A simple approach:
```json
// vercel.json
{ "crons": [{ "path": "/api/health", "schedule": "*/5 * * * *" }] }
```
Or an external uptime monitor hitting /api/health, /api/network/status, /api/idiot/browse every 5 minutes.

### 3. Investigate /api/validators/discover regression (Priority: MEDIUM)
376ms median vs 68ms baseline is a 5.5x regression. Check recent changes to:
- `web/app/api/validators/discover/route.ts`
- `web/lib/bt-metagraph.ts`
Likely cause: additional data being fetched or computed per validator.

### 4. Update Validator UID 189 (Priority: MEDIUM)
Running v728, which is 151 versions behind UID 2's v879. Should be updated to ensure consistent behavior across the validator set.

### 5. Fix ux-timing-test.mjs validator discovery URL (Priority: LOW)
The script gets HTML instead of JSON from the discovery endpoint. Likely hitting the wrong URL pattern.

## Vercel Rate Limiting Note

After approximately 20 curl API calls and 4 Playwright page loads in rapid succession, Vercel's security system started returning 403 for all endpoints. The 403 responses contain a 33KB HTML challenge page. This does not affect normal user traffic but limits automated testing frequency. Consider:
- Allowlisting the test runner's IP in Vercel Firewall
- Reducing Vercel Attack Challenge sensitivity for API routes
- Adding appropriate delays between automated test runs
