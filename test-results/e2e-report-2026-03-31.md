## Critical Issues (blocks launch)

### C-1: Vercel bot protection blocks signal detail pages
Vercel's Attack Challenge Mode aggressively blocks users who navigate quickly between pages. After ~6 rapid page loads, the entire IP is 403'd for 30+ seconds. The challenge JS (`challenge.v2.min.js`) fails to auto-resolve in Playwright, and likely fails for users with strict browser extensions.

**Impact**: Users clicking "View Signal" from browse page hit Vercel security checkpoint instead of the purchase page. Blocks the entire purchase flow.

**Fix**: Reduce Vercel Attack Challenge sensitivity in Vercel Dashboard > Project Settings > Security. Consider switching from "Challenge" to "Managed" mode, or allowlisting the /idiot/signal/* path pattern.

### C-2: Purchase MPC error messaging is misleading (FIXED)
When MPC availability check fails, the user sees "Signal not currently available. The pick may have gone stale. Check back later." even when the real problem is validator infrastructure.

**Status**: Fixed in this session. Error messages now show the actual failure reason (MPC failure, validator count, timeout, etc.). Auto-retry with fresh validator discovery added.

## Medium Issues

### M-1: Leaderboard shows empty skeleton (FIXED)
Leaderboard table displays skeleton shimmer animation but never resolves to actual data. Root cause: subgraph query filters `totalAudits_gt: 0` but no audits have been settled yet on testnet.

**Status**: Fixed. Changed filter to `totalSignals_gt: 0` so geniuses appear as soon as they create signals. Also fixed loading state to start as `true` to avoid flash of empty content.

## Minor Issues

### L-1: Browse signals takes ~8s to load on production
Browse page initially shows "Loading..." for 5-8 seconds before signals appear. The server-side API (`/api/idiot/browse`) does chunked RPC queries across 200k+ blocks.

**Impact**: Poor first impression. Users may think the page is broken.

**Recommendation**: Add a more informative loading skeleton (similar to leaderboard). Consider caching the browse results server-side for longer than 60s.

### L-2: Console warnings about subgraph network errors
Subgraph queries occasionally fail with network errors (seen in console on browse and leaderboard pages). The code handles this gracefully (falls back to empty data) but the console warnings are noisy.
